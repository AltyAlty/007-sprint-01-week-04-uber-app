Скрипты:
1. pnpm watch: компиляция файлов TypeScript из папки "src" в папку "dist" в режиме наблюдения (-w).
2. pnpm dev: запуск приложения с помощью nodemon, следя за изменениями в скомпилированном коде в папке "dist".
3. pnpm jest: запуск тестов с помощью Jest.
4. pnpm format: форматирование кода с помощью Prettier.
5. pnpm lint: запуск проверки кода с помощью ESLint и автоматического их исправления.
6. pnpm build: компиляция файлов TypeScript из папки "src" в папку "dist", нужно для деплоя на vercel.com.
7. pnpm start: запуск приложения, нужно для деплоя на vercel.com.

Файл "tsconfig.json": ручная настройка.
Файл "jest.config.json" создан вручную. Запускаются тесты только из папки "__tests__/e2e/".

Файл "tsconfig.test.json" создан вручную. Содержит отдельные правила для тестов.
1. Берет все базовые настройки из файла "tsconfig.json".
2. TypeScript проверяет типы, но не генерирует файлы .js.
3. Корнем проекта для этого конфига становится папка проекта (где лежит tsconfig). Это позволяет включать и папку "src",
и папку "__tests__", даже если они на одном уровне, и убирает конфликт вида “файл вне rootDir”.
4. В тестовый проект входят весь код приложения из папки "src" и все тесты из папки "__tests__".

Файл "eslint.config.mjs" создан вручную. Правила в этом файле:
1. Игнорирует папки "node_modules" и "dist".
2. Использует TS парсер.
3. Использует TS плагин и Prettier плагин.
4. Проверяет только файлы .ts
5. Показывает ошибку если форматирование не по правилам Prettier.
6. Показывает предупреждение для неиспользуемых переменных.
7. Показывает ошибку если используется == или !=.
8. Отключает правила ESLint, которые конфликтуют с Prettier.

Файл ".prettierrc" создан вручную. Правила в этом файле:
1. В конце ставятся точки с запятой.
2. Используются одинарные кавычки вместо двойных.
3. Используются запятые при перечислении, в том числе в конце.
4. Перенос строк длиннее 120 символов.
5. Отступ - 2 пробела.

Файл ".prettierignore" создан вручную. Правила в этом файле: Prettier игнорирует папки "node_modules" и "dist", а также
файл "pnpm-lock.json".

Jest - тестовый фреймворк для JavaScript, который поддерживает тестирование кода. Используется для написания и
выполнения юнит-тестов, тестов интеграции и моков.

TS-Jest - преобразователь для Jest, позволяющий тестировать TypeScript-код. Позволяет Jest работать с TypeScript,
компилируя его во время тестирования.

Supertest - библиотека для e2e тестирования HTTP-запросов. Используется для тестирования API, отправляя запросы к
серверу и проверяя ответы.

ESLint - линтер для JavaScript и TypeScript. Используется для анализа кода на наличие ошибок и проблем с качеством,
таких как несоответствие стилю кодирования или потенциальные ошибки.

Prettier - форматировщик кода. Используется для автоматического форматирования кода по установленным правилам, улучшая
читаемость и единообразие.

eslint-config-prettier - конфигурация для ESLint, отключающая правила, которые могут конфликтовать с Prettier. Помогает
интегрировать Prettier с ESLint без конфликтов, отключая проверки, которые дублируют функциональность Prettier.

eslint-plugin-prettier - плагин для ESLint, который интегрирует Prettier в процесс линтинга. Обеспечивает использование
Prettier как ESLint-правила, что позволяет линтить код и форматировать его в одном процессе.

swagger-jsdoc - инструмент для генерации документации API в формате Swagger на основе JSDoc комментариев в коде. Он
помогает автоматически генерировать документацию API в стандартном формате (Swagger), что упрощает поддержку и
обновление документации по мере изменения кода.

swagger-ui-express - среда для Express, которая позволяет отображать документацию Swagger в красивом веб-интерфейсе. Она
предоставляет веб-интерфейс, где разработчики могут просматривать и взаимодействовать с документацией API, что позволяет
тестировать API прямо из браузера.

Dotenv - это модуль, не зависящий ни от одного компонента, который загружает переменные окружения из файла ".env" в файл
"process.env".

/*--------------------------------------------------------------------------------------------------------------------*/

"process" - это глобальный объект в Node.js, который содержит информацию о текущем процессе выполнения. "process.env" -
это объект, содержащий все переменные окружения, доступные приложению. Переменные окружения - это значения, которые
могут быть установлены на уровне операционной системы или приложения. Переменные окружения могут использоваться для
настройки поведения программного обеспечения (например, пароли, ключи API, пути к файлам и так далее).

/*--------------------------------------------------------------------------------------------------------------------*/

Метод "sendStatus()" используется для отправки только статуса без тела ответа.

/*--------------------------------------------------------------------------------------------------------------------*/

Data Transfer Object - это объект, определяющий форму и структуру данных, которыми обмениваются разные части приложения
или разные сервисы.

/*--------------------------------------------------------------------------------------------------------------------*/

Слой Repository - это один из паттернов проектирования, позволяющий разделить логику взаимодействия с базой данных от
бизнес-логики. Это дает следующие преимущества:
1. Чистота и структура кода: роутеры избавляются от логики работы с базой.
2. Легкость тестирования: репозиторий можно тестировать независимо от остальных частей приложения.
3. Гибкость: если структура или тип БД изменится, все изменения нужно будет сделать только в репозитории, не затрагивая
другие части приложения.

Basic authorization - это способ аутентификации в веб-приложениях, при котором клиент передает серверу логин и пароль в
виде строки, закодированной в формате base64. Этот метод не является безопасным, если не используется с протоколом
HTTPS. Порядок работы этого способа примерно такой:
1. Клиент при отправке запроса должен указать в заголовке HTTP следующее:
Authorization: Basic <base64-encoded-credentials>
2. Чтобы получить строку для Basic авторизации, нужно соединить логин и пароль через двоеточие, а затем закодировать
результат в base64.
3. Сервер извлекает данные, декодирует их и проверяет логин и пароль. Если они верны, то сервер разрешает доступ.

Создание БД MongoDB локально (архив MongoDB + архив СУБД MongoDB Shell):
1. Скачать MongoDB в формате .zip с сайта mongodb.com/try/download/community.
2. Разархивировать архив в отдельную папку.
3. Запустить локальный сервер БД при помощи файла "mongod.exe".
3.1 Открыть окно терминала или окно PowerShell в папке "bin".
3.2 В папке "bin" выполнить команду "./mongod.exe --dbpath .\data\db", где ".\data\db" это путь, где мы хотите создать
саму БД (папки должны быть уже созданы). В нашем случае используется такие команды:
PowerShell:
./mongod.exe --dbpath ..\..\..\..\..\DevProjects\IT-IncubatorBackEndProjects\006-sprint-01-week-03-bloggers-app-homework\src\db\mongodb\data\db
Командная строка:
mongod --dbpath ..\..\..\..\..\DevProjects\IT-IncubatorBackEndProjects\006-sprint-01-week-03-bloggers-app-homework\src\db\mongodb\data\db
4. Запустить СУБД MongoDB Shell.
4.1 Скачать MongoDB Shell в формате .zip с сайта www.mongodb.com/try/download/shell.
4.2 Файл "mongosh.exe" из архива скопировать в папку "bin".
4.3 В папке "bin" выполнить команду "./mongosh.exe" в PowerShell, или команду "mongosh" в командной строке.
